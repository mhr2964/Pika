## Later Items — Explicitly Non-Blocking for This Launch
- Dedicated post-launch hardening beyond the documented minimum launch auth model is **non-blocking** if the must-fix launch proofs above land.
- Additional abuse sophistication beyond a basic effective throttle/control layer is **non-blocking** for this launch; the blocker is the absence of any evidenced protection on public room-entry surfaces.
- Broader secrets-operations maturity such as rotation runbooks is **non-blocking this round** if launch-critical secret definition and boot-time validation are implemented and documented.
- Share/invite channel abuse controls outside the product-controlled room-entry surfaces are **non-blocking** in this read because current evidence is scoped to the app/repo artifacts, not external platform enforcement.

## CTO Sequencing Guidance
Sequence this work to avoid churn by locking the auth/session contract first, then wiring enforcement once: (1) auth and backend jointly finalize the concrete identity model for anonymous participation, account upgrade, and signed expiring credentials; (2) backend implements route-level protection and controller/storage identity binding against that fixed contract; (3) backend/auth add the negative-path tests that prove unauthorized, forged, expired, replayed, and cross-user upgrade attempts fail; (4) in parallel, backend/devops add required secret/env definitions and boot validation; (5) once protected routes are stable, add abuse controls to room create/join and other public entry surfaces so the throttling layer is attached to the final endpoint map instead of being reworked after auth changes.
